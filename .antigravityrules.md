# Antigravity Rules - Odontologia Uteich

Sempre siga estas diretrizes arquiteturais ao trabalhar neste repositório:

## 1. Estrutura de Arquivos e Divisão de Responsabilidades
- **Schemas de Validação (Zod)**: Devem sempre ser colocados no diretório `src/schemas/` (ex: `src/schemas/plano-tratamento.ts`). Use-os tanto para validação no cliente quanto nas rotas de API.
- **Tipagens TypeScript**: Devem sempre ser colocadas no diretório `src/types/dashboard/` (ex: `src/types/dashboard/plano-tratamento.ts`).
- **Serviços de API/Server Actions**: Devem ser colocados no diretório `src/services/` (ex: `src/services/plano-tratamento.ts`). Use a diretiva `"use server"` e encapsule as requisições HTTP ou interações com o banco.
- **Rotas de API**: Devem estar sob `src/app/api/` e devem usar os schemas Zod criados para validar os dados recebidos.
- **Next.js Suspense em Server Components (Páginas)**: Todo Server Component padrão exportado sob `src/app/` que execute tarefas síncronas bloqueantes ou acesse dados dinâmicos de navegação (como `getServerSession`, `cookies()`, `headers()`, `checkAdminApi()` ou conexões diretas do Prisma sem cache) **deve** usar `<Suspense>` com um componente de Skeleton/fallback apropriado. Nunca coloque chamadas bloqueantes diretamente na página principal sem Suspense, pois isso causa erro de build no Next.js (Blocking Route). Crie um subcomponente `<Content />` para realizar a busca de dados e envolva-o em `<Suspense>` na página principal.

## 2. Padrões de Interface Gráfica (UI/UX)
- Sempre implemente telas visualmente ricas, modernas e responsivas, utilizando o sistema de design atual (cores curadas do Slate/Blue, bordas suaves `rounded-xl` ou `rounded-2xl`, e fontes elegantes).
- **Sem Placeholders**: Use dados mockados ou reais ao invés de dados estáticos sem sentido.
- **Performance**: Modais e tabelas não devem causar lag ou travamento de FPS. Utilize componentes de controle independentes com estados locais para evitar renderização desnecessária da árvore completa.
- **Modais / Dialogs**: Sempre utilize o padrão não controlado (uncontrolled hybrid pattern) usando `Dialog`, `DialogTrigger` e `DialogContent` diretamente da biblioteca de UI, sem usar estados manuais de controle de abertura (`open`/`onOpenChange`) a menos que estritamente necessário. Sempre forneça um `DialogTitle` para fins de acessibilidade.

## 3. Segurança e Auditoria
- Proteja todas as rotas administrativas chamando `checkAdminApi()` ou `requireAdminContext()` no servidor.
- Use `withAudit` nas ações mutativas (POST, PUT, DELETE) nas rotas de API para registrar logs de auditoria se aplicável.
## 4. Next.js 16 e Convenções do Projeto

* Este projeto utiliza **Next.js 16**.

* O recurso **Cache Components** está habilitado (`cacheComponents: true`).

* Ao gerar páginas, layouts, componentes, rotas de API ou Server Actions, sempre considere os padrões e restrições do Next.js 16.

* Sempre utilize as APIs e padrões mais recentes compatíveis com Next.js 16.

* Não utilize soluções depreciadas (deprecated).

* Ao trabalhar com parâmetros dinâmicos (`params`, `searchParams`) siga o comportamento do Next.js 16.

* **Todas as páginas (`page.tsx`) devem ser Server Components.**

* Não utilize `"use client"` em páginas. Caso seja necessário estado, interatividade, hooks do React ou APIs do navegador, extraia a funcionalidade para componentes específicos de cliente.

* Sempre mantenha a maior parte da árvore de renderização possível no servidor.

* Componentes de interface devem seguir a arquitetura Server First.

* O recurso de cache e renderização do Next.js 16 deve ser respeitado em toda implementação.

* **Nunca adicione comentários no código gerado.**

* Não utilize comentários de linha (`//`) ou comentários de bloco (`/* */`) em componentes, hooks, serviços, schemas, rotas de API, Server Actions ou qualquer outro código produzido para este repositório.

* O código deve ser limpo, autoexplicativo e seguir os padrões arquiteturais definidos nesta documentação sem depender de comentários para entendimento.

## 5. Padrão para Componentes Shadcn/UI

* Utilize os componentes da biblioteca Shadcn/UI da forma nativa sempre que possível.

* Não crie estados manuais para controlar abertura e fechamento de componentes que já possuem esse comportamento internamente.

* Evite utilizar propriedades `open`, `defaultOpen` e `onOpenChange` em `Dialog`, `Sheet`, `Popover`, `DropdownMenu`, `Collapsible` e componentes similares, exceto quando existir uma necessidade técnica real e justificada.

* Para diálogos, utilize preferencialmente:

  * `Dialog`
  * `DialogTrigger`
  * `DialogContent`
  * `DialogHeader`
  * `DialogTitle`

* O padrão preferencial é o comportamento nativo da biblioteca utilizando `DialogTrigger` ao invés de estados React para abrir e fechar modais.

* Sempre forneça um `DialogTitle` para acessibilidade.

* Evite complexidade desnecessária ao redor de componentes que já possuem gerenciamento interno de estado.
## 6. Organização e Tamanho dos Arquivos

* Evite arquivos excessivamente grandes.

* Sempre que uma página, componente ou módulo começar a concentrar múltiplas responsabilidades, extraia partes menores para componentes, hooks, serviços ou utilitários específicos.

* Priorize componentes pequenos, reutilizáveis e de responsabilidade única.

* Não concentre toda a lógica de uma funcionalidade dentro de um único arquivo.

* Tabelas, formulários, filtros, modais, cards, gráficos e blocos visuais complexos devem ser extraídos para componentes dedicados quando fizer sentido.

* Lógicas de transformação de dados, validações e regras de negócio não devem ficar dentro dos componentes de interface quando puderem ser isoladas.

* Hooks customizados devem ser utilizados para encapsular estados e comportamentos reutilizáveis.

* Busque manter arquivos fáceis de navegar e revisar durante code review.

* Sempre prefira composição de componentes ao invés de criar arquivos monolíticos.

### Limites Recomendados

* `page.tsx`: preferencialmente até 200-300 linhas.
* Componentes de UI: preferencialmente até 200 linhas.
* Server Actions e Services: dividir quando houver múltiplas responsabilidades.
* Evite arquivos acima de 500 linhas, salvo situações excepcionais e justificadas pela arquitetura.

### Estrutura Esperada

Ao criar uma funcionalidade nova, prefira estruturas semelhantes a:

```text
src/
├── app/
├── components/
│   └── feature/
│       ├── tabela.tsx
│       ├── formulario.tsx
│       ├── filtros.tsx
│       └── dialog.tsx
├── services/
├── schemas/
├── hooks/
├── types/
└── lib/
```

* Componentização excessiva é preferível a arquivos gigantes.

* Sempre pense primeiro em separação de responsabilidades antes de adicionar mais código a um arquivo existente.

* Antes de expandir um arquivo já grande, avalie se a nova funcionalidade pode ser implementada em um componente separado.
